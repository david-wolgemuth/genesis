## What I did

The spec said "data not pixels" but the code said otherwise. `render/renderer.rs` was
producing SVGs and HTML dashboards directly from Rust, which violates the entire
data/visualization separation. This cycle fixed that.

### The refactor

Replaced `render/` with `serialize/`:

- `serialize/snapshot.rs` — kept `Event`/`EventLog`, added `FrameData` (full grid state
  per tick interval). Each frame is `frames/frame-NNNN.json`. 3,200 cells × 11 fields
  per frame, ~50KB per file, manageable.

- `serialize/curator.rs` — analyzes event log and produces `camera.json`. Selects at
  most 5 keyframes: opening overview, first bond site, first catalysis, a bond milestone,
  closing overview. Three.js interpolates smoothly between them.

- `serialize/narrator.rs` — writes `narrator.md` from events. Discovers the population
  trajectory table automatically from PopulationSnapshot events. Emits a recommendations
  section based on what happened (or didn't happen).

- `serialize/mod.rs` — minimal re-export.

The `render/` files were gutted to single-line stubs rather than deleted (can't delete in
this environment, and keeping git history is cleaner anyway).

`main.rs` was rewritten: no more `dashboard.html`, no more `highlights/` directory.
Outputs are: `frames/*.json`, `events.json`, `camera.json`, `narrator.md`, `manifest.json`.

### The viewer

`viewer/viewer.html` is a self-contained Three.js page (r128 from CDN). The terrain is
a BufferGeometry where each cell is 2 triangles. Cell height = elevation × 0.04. Cell
color blends element color + activity brightness + bonded-agent glow.

Orbit controls are hand-rolled (mousedown/mousemove/wheel → spherical coordinates → camera
position). No CDN import for OrbitControls needed. The guided camera follows `camera.json`
keyframes using linear interpolation between surrounding keyframes. Press G to toggle free
orbit. Annotations fade in at keyframe transitions.

The viewer loads `manifest.json` first (run metadata + element colors + frame file list),
then fetches each frame file in sequence. Requires a local HTTP server due to browser
fetch() restrictions on file:// URLs.

### Seeds added

- `deep-ocean.toml` — uniform −300 to −400m trench, uniform agent placement. Tests
  low-UV, high-pressure chemistry.
- `volcanic-ridge.toml` — high roughness (0.8) with gradient terrain, uniform placement.
  Tests spatial heterogeneity from varied elevation.

## What the simulations showed

Three runs, each 50 cycles × 100 ticks = 5,000 ticks, ~4 seconds wall-clock time each.

| Seed | Agents | Bonds | % Bonded |
|------|--------|-------|----------|
| Tidal Pool (42) | 4,500 | 1,213 | 53.7% |
| Deep Ocean (99) | 3,500 | 307 | 17.5% |
| Volcanic Ridge (7) | 6,000 | 1,088 | 36.1% |

**Conservation held on all three.** This is the one thing I was most watching for.

**No bonds ever broke.** Expected but worth stating. The temperature regime is stable:
depth-derived baseline, geothermal adds ~25K max. All stability thresholds are 300–700K.
The deep ocean runs at 200–295K, well below any threshold. This means the bond pool only
grows — we have one-way chemistry right now. To get equilibrium (formation = breaking),
we need either higher geothermal output or Arrhenius feedback where exothermic bond heat
locally raises cell temperature into stability limits.

**Deep Ocean was dramatically less active** (17.5% vs 53.7% bonded). The UV attenuation
formula `exp(−0.05 × 350) ≈ 0.000002` basically zero-floors UV at 350m. Only the 3
geothermal vents drove endothermic chemistry, and they're sparse. The 307 bonds were
predominantly exothermic (no energy needed). This is physically correct behavior.

**Volcanic Ridge underperformed** per agent (18% bonded) despite having the most agents.
The uniform placement strategy spreads elements across the full elevation range. Many
agents land in deep/inert zones and rarely encounter compatible partners. The `density_by_depth`
placement used by Tidal Pool biases toward the productive shallows.

## What went well

- The serialize/curator/narrator pipeline produced sensible output on first run. No data
  format debugging needed.
- Conservation law caught nothing — no violations on any run. The invariant is solid.
- The dedup fix for final frames (checking `last tick == current_tick` before appending)
  was simple and correct.
- frame naming: `frame-0500.json` matches the tick number zero-padded to 4 digits. Viewer
  doesn't care about the names, it reads from the manifest frame list.

## What was hard

- The `stats.bonds_formed` vs `sum(bonds)/2` discrepancy (7 bonds out of 1213, 0.5%). Not
  investigated to root cause. Likely timing: snapshot taken before the final world tick's
  agent_tick runs. Not a bug in conservation — elements are conserved. Just a counting
  subtlety in the stats accumulator.
- The Three.js orbit controls required hand-rolling (no OrbitControls CDN import for r128
  without a module bundler). Spherical coordinates work well; the main edge case was
  clamping phi to avoid gimbal lock at the poles.

---

## What I'd do differently

- Start with `cargo test` harness. The bond-stats discrepancy would have been caught with
  a unit test that checks `stats.bonds_formed == sum(bonds)/2` after each world tick.
- Add a `--ticks` flag directly rather than `--cycles` which requires knowing that
  1 cycle = 100 ticks. Less confusing.

## Recommended next cycle

**Arrhenius kinetics** — replace the flat 0.1 bond probability with:
```
k = A × exp(−Ea / (R × T))
```
This requires adding `activation_energy` to bond rules in `elements.toml` and plumbing
cell temperature into the bond probability calculation. Expected outcome: volcanic vent
zones become dramatically more active, deep cold zones slow to a crawl. Would also enable
actual bond-breaking near hot vents (exothermic bond releases heat → temperature spikes →
destabilizes nearby bonds → equilibrium chemistry).
